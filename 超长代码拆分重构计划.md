## 统一拆分守则
- 单文件目标：控制在 `150-200` 行附近；复杂页面、服务和仓储允许存在聚合入口，但入口不应再承载细节实现。
- 单函数 / 单 Hook 目标：控制在 `30` 行附近；超出后优先抽纯函数、子状态机或 DTO / presenter mapper。
- 前端拆分模板：`PageShell` / `Filters` / `TableOrList` / `Dialogs` / `useXxxPage` / `types` / `api`。
- 后端拆分模板：`facade/orchestrator` / `validator` / `adapter` / `calculator` / `mapper` / `query` / `presenter`。
- 生成物治理：`backend/ent/*`、`frontend/src/generated/*`、大型 seed / registry 文件都要回到 schema、generator 或原始数据切片层处理。
- 每波拆分都要补最小验证：前端 `typecheck + 关键组件测试`，后端 `go test` 至少覆盖被拆模块主链路。
- **拆分豁免**：以下类型文件不适用 150-200 行阈值，不纳入拆分清单：
  - **生成物**：`wire_gen.go`、`backend/ent/*`、`frontend/src/generated/*` 等自动生成文件，不手改。
  - **已落地的 SQL migration**：已执行的迁移脚本拆分既危险又无意义，仅对未来新迁移做预防。
  - **声明式配置文件**：路由声明（`router/index.ts`）、全局样式（`style.css`）、示例配置（`config.example.yaml`）等天然扁平声明式，拆分增加导航成本但不降低认知复杂度。
  - **纯文档**：README、VISUAL_GUIDE.md、USAGE_EXAMPLES.md 等拆分会降低可读性和可发现性。
  - **部署脚本**：`install.sh` 等追求单文件可移植执行，拆分引入 source 复杂度。
  - **JSON 数据文件**：`preset_overlay.json` 等归 P3 上游数据治理，不做手工拆分。
  - **依赖锁文件**：`pnpm-lock.yaml` 永远不该手动拆。
  - **测试文件**：测试天然冗长（fixture + 断言），除非单文件 >2000 行，否则不强制拆分。
- **防止过度碎片化**：拆出的文件应有足够的独立语义；<50 行的微型组件若只有一个消费者、<30 行的微型 util 应优先合并到已有共享模块，避免文件爆炸。
